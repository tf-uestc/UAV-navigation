## 2. 逐模块分析

### 2.1 整体架构（ARCHITECTURE.md）

**✅ 设计正确。**

- 进程间用 ROS topic 解耦，进程内用 Python ABC 解耦——分层清晰
- TF 帧约定（REP-103 ENU + REP-105 命名）符合 ROS 标准
- 迁移策略合理：老 launch 文件保留，新链路用新的 v2/v3 launch，可随时回退
- `setpoint_bridge.py` 零改动的承诺是兑现的

### 2.2 F2 — VLM Grounding 感知

**✅ 可行。**

从"文本返回 (u,v)"升级到 bbox grounding 是正确方向。Qwen-VL 的 grounding API 已成熟，中位数深度融合（替代单点 3×3 邻域）对外点更鲁棒。

**关注点：**

| # | 问题 | 严重度 | 建议 |
|---|------|--------|------|
| 1 | 坐标归一化判定（1000 归一化 vs 像素）在 848×480 分辨率上有边角情况 | 🟡 中 | spec 已提"第一次肉眼确认"，补充：**首次启动自动 log bbox 原始坐标范围**，便于事后诊断 |
| 2 | `detector_args` 注入方式未明确定义 | 🟡 低 | plan 说"字典注入"，统一为 `**kwargs` 传给 `__init__`，在 plan.md 里写死约定 |
| 3 | API Key 当前硬编码在 `vlm_navigation.launch` 第 43 行 | 🔴 高 | **不等 F2，立即修**。从 launch 删除，改为 `$(env VLM_API_KEY)`，启动前 `export` 注入 |

**其他做得好的地方：**
- bbox 内深度有效率 ≥ 30% 才接受，避免透明/反射表面误判
- confidence 用"框内有效深度比"作 proxy，务实
- 单帧检测 + 失败返回 None，上层可重试，不阻塞飞控链路

### 2.3 F3 — TF 链与坐标系修正

**✅ 可行，思路正确。**

把当前 `camera_to_world_approx` 里的 yaw-only 近似替换为完整 tf2 变换链，是消除俯仰/横滚姿态下定位误差的正确做法。动捕集成路径对下游透明，设计干净。

**关注点：**

| # | 问题 | 严重度 | 建议 |
|---|------|--------|------|
| 1 | mavros TF 发布机制需确认 | 🟡 中 | plan 说改 `px4_config.yaml` 的 `local_position.tf.send`，但 mavros 不同版本 TF 发布方式不同（有的是 plugin，有的是 yaml 参数）。**先跑 `rostopic echo /tf` 看现状有没有 `map→base_link`，再决定改什么。** 目前 mavros 可能已经在发了 |
| 2 | 用 image `header.stamp` 做 tf lookup | 🟢 亮点 | 这是正确的做法，避免 50ms 网络延迟带来的姿态错位 |
| 3 | TF 健康检查脚本 `tf_healthcheck.py` | 🟢 亮点 | 启动时硬失败比运行时静默错误好得多 |

### 2.4 F4 — EGO-Planner 集成

**✅ 方向正确。**

EGO-Planner 作为局部避障规划器替代当前的"直线缩进"是正确的技术选择。Planner ABC 抽象为后续论文对比（SUPER、A* 等）留了空间。

**关注点：**

| # | 问题 | 严重度 | 建议 |
|---|------|--------|------|
| 1 | EGO 输出 msg 类型未知（`Bspline` vs `PositionCommand` vs `Path`） | 🟡 中 | 这是 Week 13 "摸接口"的任务。确认后告诉我，adapter 实现方法不同 |
| 2 | `trajectory_follower` 时间戳 hack | 🟡 低 | plan 说把 `t_from_start` 塞进 `header.stamp` 偷渡时间戳。技术上行得通但语义不干净。如果轨迹等间隔，直接按 index 线性走更简单；如果不等间隔，在 `TrajectoryPoint` 里显式存 `t_from_start` |
| 3 | F2→goal_pose 降级路径的类型不匹配 | 🔴 高 | 见下文 §3.2 |
| 4 | EGO 集成风险（话题名/frame_id 不匹配） | 🟡 中 | Week 13-14 安排充分（4 周），已跑通官方 demo 是一个好的开端。如果 Week 14 还接不上，plan 里的降级方案（退到静态地图 + 直线）是可接受的 |

#### 关于 ego-planner 是否必须移出 workspace

spec 建议移出的理由（编译隔离、依赖污染、构建速度）在**新搭建时**是成立的。但你的情况是：
- 已经在 `ws/src/` 下**编译成功**
- 已经**跑通 demo**
- 你没有计划修改 EGO 源码（只是 wrap 它的 ROS 接口）

**结论：不需要移出。** 既然已经工作且你不会改动 EGO，强行迁移反而引入新风险（路径、环境变量、source 顺序）。建一个 `EGO_WORKSPACE` 变量指向当前 `ws/` 根即可，adapter 依然按计划写，不依赖 ego 在哪个目录。

## 3. 需要修正的硬伤

### 3.1 API Key 泄露（优先级：立即）

**位置**：`ws/src/uav_vln_bringup/launch/vlm_navigation.launch` 第 43 行

```xml
<param name="vlm_api_key" value="sk-xxx" />
```

**修复**：
```xml
<param name="vlm_api_key" value="$(env VLM_API_KEY)" />
```

启动前 `export VLM_API_KEY=sk-xxx`。注意检查 git 历史里是否已有这个 key，如果已 commit 需要清理。

### 3.2 F2→goal_pose 降级路径的类型不匹配

**当前设计矛盾**：
- F2（`detector_node`）输出 `/uav/target_world`，类型是 `PointStamped`
- `setpoint_bridge` 吃的是 `/uav/goal_pose`，类型是 `PoseStamped`
- F4 未接入时，需要一种方式把 `PointStamped` 喂给 setpoint_bridge

**推荐方案**：在 launch 里用一个 nodelet 做类型桥接（不改任何 Python 代码）：

```xml
<node pkg="topic_tools" type="transform" name="target_to_goal"
      args="/uav/target_world /uav/goal_pose geometry_msgs/PoseStamped 
            'm.pose.position.x = m0.point.x; m.pose.position.y = m0.point.y; 
             m.pose.position.z = m0.point.z; m.pose.orientation.w = 1.0'"/>
```

或者更简单：让 `detector_node` 同时直接发 `/uav/goal_pose`，F4 接入后在 launch 里关掉这个转发。

## 4. 实施顺序建议

F3（TF 诊断）先于 F2，因为 F2 的世界坐标转换依赖 TF 链就绪：

```
Week X:      F3 诊断阶段（跑 rqt_tf_tree + rostopic echo /tf，摸清现状）
Week X+1:    F2 开始实施（与 F3 并行——F3 补齐 TF 配置，F2 写 ABC + detector_node）
Week X+2:    F2 完成验收（用 dummy detector 跑通骨架，切 QwenVLGroundingDetector 验收指标）
Week X+3-4:  F4 Week 13-14（摸 EGO 接口 + ABC + dummy 端到端 + 替换为 EgoPlannerAdapter）
Week X+5-6:  F4 Week 15-16（真实深度接入 + 避障飞行 + 评测）
```

**最早可并行**：F2 和 F3 的 TF 补全可以在同一周做。F3 诊断出来后，F2 和 F3 各自推进。

## 5. 总体风险评估

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| EGO 话题/msg 类型不兼容 | 中 | 高 | Week 13 "摸接口" + Week 14 有降级方案 |

| 坐标系归一化误判导致框错位 | 中 | 中 | 首次运行必须看 `/uav/target_debug` 目视确认 |
