# 基于视觉语言模型的无人机语言交互与自主导航（仿真优先 + EGO-Planner）计划书

> 适用范围：硕士/毕设项目计划（20 周）。
> 
> 核心原则：**先闭环、后增强；先仿真、后实机；先可复现、后追性能**。

---

## 1. 项目概述

### 1.1 题目
基于视觉语言模型的无人机语言交互与自主导航系统设计与实现（仿真优先）。

### 1.2 总目标
实现无人机在 ROS + Gazebo/PX4 SITL 仿真中：
1) 接收自然语言指令；
2) 解析为结构化任务（JSON）；
3) 通过视觉/深度感知定位目标；
4) 基于 **EGO-Planner** 生成动力学可行轨迹并避障；
5) 通过 MAVROS/Offboard 控制 PX4 执行并完成任务。

### 1.3 研究问题与工程闭环
- **语义到几何**：将“救援目标/物资箱/障碍”等语义概念稳定转换为可规划的 3D 坐标与约束。
- **实时规划**：在部分未知/动态障碍场景中进行局部重规划，输出可飞轨迹。
- **系统集成**：ROS 多节点低延迟通信、可复现评测与记录。

## 2. 总体技术路线（与开题对齐）

### 2.1 系统分层
1. **自然语言理解层（LLM）**
   - 输入：中文自然语言指令
   - 输出：结构化 JSON（目标类型/属性/动作/约束）

2. **视觉感知层（VLM/检测 + 深度/真值辅助）**
   - 输入：RGB + Depth（仿真深度） + 目标描述
   - 输出：目标 2D 框/像素点 + 目标 3D 点（相机系/世界系）+ 置信度

3. **轨迹规划层（EGO-Planner）**
   - 输入：当前状态、占据/ESDF/点云、目标点/约束
   - 输出：动力学可行轨迹（位置/速度/加速度/偏航）与局部重规划

4. **控制执行层（PX4 SITL + MAVROS）**
   - 输入：期望位姿/轨迹设定点（Offboard）
   - 输出：无人机执行 + 状态反馈

### 2.2 “最小可交付版本（MVP）”定义
MVP 必须满足：
- 在 Gazebo 仿真场景中，通过一句自然语言指令触发任务；
- 系统完成“目标定位 → 规划 → 飞行 → 到达目标半径内悬停/降落”闭环；
- 有一键复现实验脚本与日志，能输出核心指标。

### 2.3 扩展项（加分但不阻塞毕业）
- 多轮澄清对话（解析失败时询问用户）
- 动态障碍情况下的重规划成功率提升
- VLM 领域自适应（微调/LoRA）
- 实机验证（在具备条件时）

## 3. 关键模块设计

### 3.1 指令结构化（JSON Schema）
建议固定 Schema，便于单元测试与统计：

- `task_type`: `search | go_to | deliver | inspect | circle`
- `target`: `{ type, attributes, text }`
- `action`: `{ mode, distance, altitude, hover_time, land }`
- `constraints`: `{ min_distance_to_obstacle, speed_limit, timeout_s }`
- `fallback`: `{ if_not_found: return_home | hover | spiral_search }`

解析策略：
- 以**可控提示词 + few-shot**为主；
- 输出必须可被 JSON parser 严格解析；
- 失败必须返回可解释错误码（便于评测）。

### 3.2 感知与坐标转换
仿真优先的落地建议：
- **第一步（MVP）**：使用 Gazebo/仿真模型提供的深度图 + 相机内参，把检测到的目标像素（或框中心）回投到 3D；
- **第二步（增强）**：多帧滤波（EMA/Kalman）提升稳定性；
- **第三步（扩展）**：对遮挡/复杂背景引入开放词汇检测（GroundingDINO/OWL-ViT）或 VLM grounding。

坐标系：
- 明确 `camera_link`、`base_link`、`map/odom` 的 TF 链；
- 目标点统一输出到 `map` 或 `odom`（与规划器一致）。

### 3.3 EGO-Planner 集成
落地要点：
- 输入地图：由深度生成点云/占据栅格/ESDF（按 EGO-Planner 接口要求选一种可跑通的最简形式）
- 轨迹输出：转为 MAVROS Offboard 可消费的 setpoint（位置/速度）
- 重规划：当目标更新或障碍更新触发 replanning

> 注：EGO-Planner 的“能跑起来”优先于“极致性能”。先用官方推荐参数与简单场景完成闭环。

## 4. 数据、场景与评测设计（必须可复现）

### 4.1 仿真场景集（Scenario Set）
至少准备 3 类场景，每类 5-10 个任务：
1. **简单场景**：稀疏障碍、单目标
2. **中等场景**：多障碍、干扰物、目标在视野边缘
3. **复杂场景（扩展）**：动态障碍/遮挡/噪声

每个任务固定随机种子与初始位姿，确保可复现。

### 4.2 指标（验收标准写死）
必须输出并保存为 CSV/JSON：

**语言理解指标**
- JSON 解析成功率（%）
- 字段正确率（按字段统计）
- 端到端解析耗时（ms）

**感知指标**
- 检测成功率（%）（是否在限定时间内给出目标）
- 2D IoU@0.5（有标注时）
- 3D 定位误差（m）：`||p_est - p_gt||`（仿真真值可取）

**规划与控制指标**
- 任务成功率（%）：在 `timeout_s` 内到达目标半径 `r`（如 1.5m）
- 平均耗时（s）
- replanning 次数与平均 replanning 时间（ms）
- 最小障碍距离（m）（安全性）

**系统指标**
- 端到端延迟：指令到起飞/第一次轨迹发布的时间
- CPU/GPU 占用（可选）

## 5. 目录结构建议（与现有仓库兼容）

建议在 `/home/tf/毕业设计` 新增或规范化：

- `docs/`：环境搭建、实验说明、复现实验步骤
- `simulation/`：Gazebo 场景、launch、PX4 参数
- `src/`：核心节点
  - `language/`：parser、prompt、测试集
  - `perception/`：detector、depth backproject、tf
  - `planning/`：EGO-Planner glue code、地图接口
  - `control/`：offboard 控制、状态机
- `experiments/`：评测脚本、日志解析、画图

> 现有 `UAV-VLA/` 可作为“参考实现/对比实验”，但毕设主线建议单独命名主工程，避免耦合过深。

## 6. 分阶段计划（20 周）

> 每阶段都要求：可运行 demo + 日志 + 简短实验记录。

### 第 1 阶段：环境与基础飞行闭环（第 1-4 周）
**目标**：ROS + Gazebo + PX4 SITL + MAVROS 完整跑通

- 安装/配置：Ubuntu20.04、ROS Noetic、Gazebo、PX4 SITL、MAVROS
- Offboard：起飞/悬停/降落、航点飞行（稳定运行 5 次以上）
- 相机话题：RGB/Depth 获取与录包（rosbag）

**产出**
- 一键启动 launch
- 基础飞控脚本（状态机）
- 记录：RTF/命令/参数文档

**验收标准**
- 仿真中连续 5 次完成起飞-悬停-降落，且无 failsafe

### 第 2 阶段：语言理解模块（第 5-8 周）
**目标**：自然语言 → JSON（可测、可统计、可兜底）

- 定义 JSON Schema + 单元测试集（>=50 条指令）
- Prompt 模板 + few-shot
- 输出错误码与 fallback 策略

**产出**
- `instruction_parser` + 测试报告
- 指令集与期望 JSON（golden data）

**验收标准**
- JSON 解析成功率 ≥ 95%
- 字段正确率（核心字段：target、action）≥ 85%（以自建标注为准）

### 第 3 阶段：感知与 3D 定位（第 9-12 周）
**目标**：给出目标的世界坐标（先稳定，再鲁棒）

- MVP：简单检测（可先用已知标记物/仿真模型名/颜色规则）+ 深度回投得到 3D
- TF 链打通：`camera_link` 到 `map/odom`
- 多帧滤波提升稳定性

**产出**
- 目标 3D 定位节点
- 定位误差统计脚本

**验收标准**
- 简单场景定位 RMSE ≤ 0.5~1.0m（阈值按场景调整并写入报告）

### 第 4 阶段：EGO-Planner 规划与控制集成（第 13-16 周）
**目标**：基于目标点与地图，输出可飞轨迹并执行到达

- EGO-Planner 在仿真中跑通（官方 demo + 你的无人机模型）
- 地图输入接口：深度 → 点云/占据/ESDF（最简可用链路）
- 轨迹输出到 Offboard setpoint
- replanning 触发与安全策略（例如目标点更新/障碍更新/超时）

**产出**
- 规划 glue 节点与配置
- 单场景端到端 demo

**验收标准**
- 中等场景任务成功率 ≥ 70%（10 次任务）
- 轨迹发布频率满足 PX4 offboard 要求（例如 >= 20Hz，按你的实现写死）

### 第 5 阶段：系统实验、对比与论文（第 17-20 周）
**目标**：完成评测体系、对比实验与论文初稿

- 批量跑场景集，自动汇总指标
- 对比基线：
  - 无语言：直接给定目标点
  - 无 replanning：只规划一次
  - A*（可选）：作为对照（若时间允许）
- 论文撰写与图表生成

**产出**
- 实验报告（表格+曲线）
- 完整论文初稿 + 演示视频

**验收标准**
- 至少 3 类场景、每类 >=5 个任务的完整统计
- 可复现：从 README 步骤可在新环境复跑得到同等格式结果

## 8. 风险与应对（兜底方案必须写清）

| 风险 | 影响 | 优先级 | 应对/兜底 |
|---|---|---:|---|
| EGO-Planner 集成成本高 | 卡在第 13-16 周 | 高 | 先跑官方 demo → 再替换无人机模型；必要时退化为静态地图 + 低频 replanning |
| VLM/检测不稳定 | 目标点抖动导致规划失败 | 高 | MVP 用仿真可控目标；多帧滤波；目标丢失进入 `fallback` 搜索/悬停 |
| 坐标系/TF 错误 | 误定位、飞偏 | 高 | 阶段 3 单独做 TF 验证工具（可视化、单元测试） |
| API 模型不可用/延迟 | 端到端延迟大 | 中 | 保留本地开源模型/规则基线；API 仅做增强对比 |
| 仿真与实机差异 | 实机不可复现 | 中 | 论文主结果以仿真为准；实机作为扩展，不影响主线 |
